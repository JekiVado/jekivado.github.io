# 首测风险台账

生成时间：2026-06-29  
分析目标：把架构、功能、资源、配置、网络和质量保障层面的风险收束成可跟踪台账，服务首次技术测试前的研发管理、QA 验收和制作人决策。

## 1. 总体结论

项目当前不是“功能没做完”的主要风险，而是“主链路能否在真机真包、真服、弱网、多玩家、资源热更条件下稳定跑通”的风险。

建议将首测风险管理目标定为：

> P0 风险必须在首测前关闭或给出明确降级方案；P1 风险必须有 owner 和验证用例；P2 风险可以带入首测，但需要记录在已知问题中。

当前最需要盯住的 P0 风险：

1. 热更 DLL/AOT 真包完整性。
2. 断线/重连/假连接。
3. 频道加入卡 loading。
4. Bugly/崩溃日志未启用或不可用。
5. UIConfig 缺失资源被误开放。
6. 首测功能入口未收敛。

## 2. 风险分级规则

| 等级 | 定义 | 处理要求 |
|---|---|---|
| P0 | 阻断启动、登录、进服、主玩法，或导致无法定位首测崩溃 | 首测前必须关闭或提供可验证降级 |
| P1 | 明显影响体验、稳定性或 QA 覆盖，会扩大 bug 面 | 必须有 owner、用例、修复计划 |
| P2 | 不阻断首测主链路，但会增加维护成本或边缘体验问题 | 记录为已知问题，排期治理 |

## 4. P0 风险详情

### R-001 热更 DLL/AOT 真包完整性不可直接确认

等级：P0  
建议 owner：客户端主程 / 构建负责人

证据：

- `LoadDll` 运行时固定加载 13 个 DLL：`Config`、`GameCore`、`GameCommon`、`Proto`、`Manager`、`DataManager`、`Character`、`Battle`、`GamePlay`、`Net`、`Scene`、`UI`、`Timeline`。
- `Assets/StreamingAssets` 中没有直接可见的同名 `.dll`，资源被打入大量哈希 `.a` 文件。
- `HybridCLRData/HotUpdateDlls/Android` 当前直接可见部分 DLL，但未直接看到 `Battle/GamePlay/Net/Scene/UI/Timeline`。
- AOT 固定列表 `System.dll`、`System.Core.dll`、`mscorlib.dll`、`Framework.dll` 未在常规目录直接扫到。

影响：

- 真机启动失败。
- 热更程序集加载失败。
- AOT 元数据缺失导致泛型/反射相关崩溃。
- 编辑器正常但 Android 包异常。

验收标准：

- 首测 Android 包启动日志逐项打印 13 个 DLL 加载成功。
- AOT 元数据逐项打印加载成功。
- CI 或出包脚本输出 DLL/AOT/AssetBundle 清单。
- 任意缺失项阻断出包。

### R-002 断线重连逻辑不完整

等级：P0  
建议 owner：客户端网络 / 服务端

证据：

- `NetworkManager.IsConnect()` 固定返回 `true`。
- `OnDisConnect()` 中断线弹窗和登出处理大多被注释。
- `ReConnect()` 中有“TODO：跳转主场景”，当前直接广播加载广场。
- 弱网相关 UI `UIPopReconnect` 存在，但链路未完整启用。

影响：

- 玩家断网后客户端仍认为在线。
- 房间/战斗状态无法恢复。
- 频道或场景 loading 无法结束。
- 首测弱网反馈会大量集中在“卡死/没反应”。

验收标准：

- 登录页断网、频道 loading 断网、广场断网、房间断网、战斗断网都有明确表现。
- 断线后不允许无限 loading。
- 重连后状态恢复或明确回到安全场景。
- 服务端踢号、维护、网络失败有一致 UI 提示。

### R-003 频道加入失败可能卡 loading

等级：P0  
建议 owner：客户端 UI/网络 / 服务端

证据：

- `ChannelProtoExtensions.Send_ChannelJoin()` 发送请求前先广播 `MN_UICHANNEL_JOIN_RESULT`。
- `UIChannel.OnJoinResult()` 收到后立即打开 loading 并关闭频道 UI。
- 真实服务器回包 `S2CChannelJoin` 后才加载广场。
- 若请求失败、超时、断线，UI 可能已经进入 loading 状态。

影响：

- 老玩家登录后卡在进频道。
- 新玩家创角后无法进入广场。
- 首测转化漏斗第一大阻断点。

验收标准：

- 频道加入有超时。
- 失败后关闭 loading，回到频道选择。
- `NetworkManager.ChannelId` 在失败时回滚或不提前写入。
- 服务端错误码能正常提示。

### R-004 崩溃采集疑似未启用

等级：P0  
建议 owner：客户端 / 运维

证据：

- `GameLaunch.Awake()` 中 `gameObject.AddComponent<BuglyInit>()` 被注释。
- 工程中存在 Bugly 插件，但当前启动链路未看到启用。
- Debug 日志较多：`Debug.Log` 约 723 处，`Debug.LogError` 约 168 处，但本地日志不能替代崩溃平台。

影响：

- 首测崩溃无法聚合。
- 设备、系统、版本、堆栈不可见。
- 修复速度大幅降低。

验收标准：

- 真机包故意触发测试异常，Bugly/等效平台能看到。
- 崩溃记录包含包版本、资源版本、设备、系统、堆栈。
- 关键登录/热更/场景 loading 日志可关联用户。

### R-005 UIConfig 存在缺失资源

等级：P0/P1  
建议 owner：客户端 UI / 资源 / 运营配置

证据：

- UIConfig 唯一 `PrefabPath`：146。
- 常规路径找到：124。
- 缺失或路径不匹配：22。
- 缺失项集中在活动、首充、礼包、抽卡、VIP、CP 商城、礼物、战斗测试。
- `UIActivityCheckIn` 是路径漂移：配置为 `UI/Activity/CheckIn/UIActivityCheckIn`，实际在 `UI/ActivityCenter/CheckIn/UIActivityCheckIn`。

影响：

- 玩家点击入口后 UI 加载失败。
- 活动/任务/红点动态打开缺失 UI。
- 商业化入口误开导致首测体验崩坏。

验收标准：

- 首测开放的每一个 UIFlag 都通过 Prefab 存在性检查。
- 隐藏功能没有主界面入口、任务跳转、活动配置、红点入口。
- `UIActivityCheckIn` 如开放签到必须修正路径。

### R-006 首测功能入口未收敛

等级：P0/P1  
建议 owner：制作人 / 策划 / 客户端 / 运营配置

证据：

- `SquareScene` 触发点可打开房间、家族、排行、商城、拍照房。
- `UIMain` 菜单可打开背包、好友、设置、邮件、成就、排行、礼物、拍照、客服。
- `MainIconDataManager` 会根据活动配置动态显示入口。
- `TaskDataManager` 任务跳转会打开背包、好友、聊天、房间等。

影响：

- 首测玩家进入未稳定系统。
- QA 反馈分散，主链路问题被外围功能淹没。
- 研发修复优先级混乱。

验收标准：

- 首测入口开关表确认。
- 客户端入口、服务端功能开关、活动配置、任务跳转一致。
- C/D 档功能隐藏或灰度。

## 5. P1 风险详情

### R-007 家族入口与协议初始化不一致

等级：P1  
建议 owner：客户端 / 服务端

证据：

- `FamilyProtoExtensions` 本身有 19 个服务端消息注册和 21 个事件请求。
- `NetworkManager` 中 `FamilyProtoExtensions.Init()` 被注释。
- `SquareScene` 触发点仍会打开 `UIFamily` 并请求家族列表。

建议：

- 首测隐藏家族入口。
- 若开放，必须恢复协议初始化并完成服务端联调。

验收标准：

- 家族入口不可见，或家族创建/列表/申请/退出全链路跑通。

### R-008 拍照馆成熟度不足

等级：P1  
建议 owner：客户端玩法 / SDK / QA

证据：

- PhotoStudio UI 体量较大，功能链路复杂。
- 代码存在“函数缺失，等补全 TODO”。
- `PhotoStudioRegistry` 中统计/背景相关处理仍有 TODO。
- 拍照依赖相册保存、分享、权限和多人房间同步。

建议：

- 首测普通玩家隐藏拍照入口。
- 内部专项灰度验证保存、分享、邀请、权限。

验收标准：

- 创建拍照房、邀请、拍摄、保存、分享、退出全部有明确结果。

### R-009 商城/支付/礼包/首充环境未确认

等级：P1  
建议 owner：SDK / 服务端 / 运营

证据：

- `JWGameSDK` 有 Android 登录、支付、分享接口。
- UIConfig 中 VIP、首充、礼包、抽卡等多个商业化 UI 缺失。
- 商城与货币、购物车、赠送、性别限制、时效道具耦合。

建议：

- 首测关闭真实支付。
- 商城只开放浏览或测试货币。
- 首充、礼包、VIP、抽卡隐藏。

验收标准：

- 支付入口不可见，或沙盒订单全链路可回放。

### R-010 活动和任务可能跳转到隐藏模块

等级：P1  
建议 owner：策划 / 客户端 / 运营配置

证据：

- `ActivityDataManager` 依赖活动时间、展示时间、奖励、抽奖券等配置。
- `MainIconDataManager` 动态显示主界面活动入口。
- `TaskDataManager` 任务跳转会打开背包、好友、聊天、房间、活动中心等。

建议：

- 首测只开放少量基础任务和简单活动。
- 任务跳转目标必须和首测开放范围一致。

验收标准：

- 不存在任务/活动引导玩家进入隐藏模块。

### R-011 自动化测试缺失

等级：P1  
建议 owner：QA / 客户端

证据：

- `Packages/manifest.json` 引入了 Unity Test Framework。
- 但源码中未扫到 `NUnit`、`UnityTest`、`TestFixture`、`Assert.` 用例。

影响：

- 每次出包依赖人工冒烟。
- 热更、资源、UIConfig、配置重复等问题容易回归。

建议：

- 先不追求完整自动化，至少补出包前静态检查和主链路冒烟清单。

验收标准：

- 每日构建自动检查 UIConfig、配置 JSON、场景、热更 DLL/AOT。
- QA 每日冒烟覆盖启动到结算。

### R-012 服务器环境硬编码为 dev 域名

等级：P1  
建议 owner：服务端 / 客户端 / 运维

证据：

- `NetworkManager` 当前 Host 为 `jw-game-api-dev.auforever.com`。
- 同文件存在多条内网 IP 注释。

影响：

- 首测环境与开发环境混用。
- 数据污染。
- 问题定位时无法区分开发服/测试服。

验收标准：

- 首测环境域名、端口、版本、数据库隔离明确。
- 客户端包能确认当前连接环境。

### R-013 资源包体和加载压力较高

等级：P1  
建议 owner：客户端资源 / 运维

证据：

- `Assets/StreamingAssets` 约 3.56 GB。
- `AssetsPackage/Animation` 约 5.84 GB，`Scene` 约 2.00 GB，`Role` 约 1.28 GB，`Sound` 约 1.11 GB。

影响：

- 首次下载压力大。
- 弱网更新失败。
- 加载时间长。

验收标准：

- 首测首包大小、热更包大小、CDN 下载速度有明确数据。
- 非开放模块资源尽量不进首包或不强制下载。

## 7. 首测前关闭标准

建议首测准入会按以下标准逐条确认：

| 类型 | 必须达成 |
|---|---|
| 启动热更 | 13 个热更 DLL 和 AOT 元数据真机加载日志齐全 |
| 登录频道 | 登录、创角、进频道失败/超时均有回退 |
| 断线重连 | 登录、频道、广场、房间、战斗断线都有明确表现 |
| 崩溃采集 | 真包崩溃能在平台看到堆栈和版本 |
| UI 资源 | 首测开放 UIFlag 全部资源存在 |
| 功能范围 | 首测入口开关表完成，C/D 档功能隐藏 |
| 环境 | 首测服务器环境和开发环境隔离 |
| QA | 每日冒烟覆盖启动到结算 |

## 8. 建议会议问题

制作人可以直接在首测准备会上问：

1. R-001 到 R-006 每一项现在 owner 是谁？预计哪天关闭？
2. 真机包启动日志是否能证明 13 个 DLL 和 AOT 元数据都加载成功？
3. 频道加入失败是否已经有超时回退？
4. Bugly 是否已在首测包启用并验证？
5. 家族、拍照、心动房、婚礼、VIP、首充、礼包、抽卡是否确认隐藏？
6. 任务和活动是否会跳转到隐藏功能？
7. 首测环境是否是独立环境，而不是 dev 域名直接复用？
8. 如果首测当天大面积卡 loading，研发如何定位到资源、网络、服务端还是 UI？

## 9. 管理建议

当前最有效的管理动作不是继续增加新功能，而是建立一个“首测准入闸门”：

- P0 未关闭，不出首测包。
- P1 未关闭，必须写入已知问题和灰度策略。
- C/D 档功能默认隐藏，除非专项 owner 明确承担风险。
- 每日构建必须产出主链路冒烟结果。
- 每天只看一个核心指标：从启动到完成一局的成功率。

客观评价：

> 这个项目的首测机会是存在的，但成功关键不是功能数量，而是风险收敛。  
> 如果主链路稳、日志可见、入口收住，首测会给出有效反馈；如果入口放开、弱网没兜住、崩溃不可见，首测很容易变成问题噪音。

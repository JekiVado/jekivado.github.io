# 《DanceNew 手游项目源码分析报告》结构草案

## 0. 报告定位与阅读方式

### 0.0 当前项目背景
- 项目阶段：研发期，尚未上线。
- 当前版本状态：主体开发基本完成，处于 Bug 修复阶段。
- 历史情况：底层和基础系统模块经历过多次改动、推翻和重构，需重点评估架构稳定性、历史包袱和返工风险。
- 近期目标：预计下个月进行首次技术测试。
- 核心关注：当前开发进展是否真实可测、首次测试前是否存在关键坑点、项目客观质量和风险等级如何。

### 0.1 报告目标
- 面向制作人、项目管理、主程、技术美术和服务端协作方，建立对当前 Unity 手游项目的共同认知。
- 从源码、资源、配置、构建与工具链出发，判断项目的架构成熟度、功能完成度、开发进度、风险点和后续排期关注项。
- 输出既能用于管理决策，也能让研发团队按源码位置复核的分析材料。

### 0.2 结论分层
- 制作人摘要：当前项目做到了什么、还缺什么、最大风险是什么。
- 首测 readiness 结论：以“能否支撑首次技术测试”为核心，给出通过/有条件通过/高风险/不建议测试的判断。
- 研发摘要：系统边界、模块依赖、启动链路、热更/资源/网络/配置等关键工程机制。
- 风险摘要：历史重构遗留、基础系统稳定性、核心链路缺陷、测试覆盖不足、构建和热更风险。
- 证据索引：每个判断对应到源码目录、配置文件、场景、资源或构建产物。

### 0.3 分析范围与限制
- 范围：Unity 客户端工程、客户端脚本程序集、场景和资源组织、配置生成、协议与网络层、SDK 接入、热更和构建链路。
- 暂不覆盖或仅做接口判断：服务端真实实现、运营后台、外部 SDK 控制台配置、线上数据表现。
- 进度判断口径：以源码完整度、场景/Prefab/UI 数量、配置表、网络协议、入口链路、资源产物、近期文件变更为主要依据，不直接等同于可上线质量。

## 1. 项目总览

### 1.1 基础信息
- 项目类型：Unity 手游客户端。
- Unity 版本：从 `ProjectSettings/ProjectVersion.txt` 提取。
- 渲染管线和平台目标：从 `Packages/manifest.json`、`ProjectSettings`、构建设置和资源目录判断。
- 版本管理状态：SVN 工程结构、工作区状态、近期修改目录。

### 1.2 技术栈概览
- Unity 2022.3 LTS / URP。
- HybridCLR 热更新。
- AssetBundle/资源更新链路。
- Luban 配置生成体系。
- UniTask、DOTween、Spine、TextMeshPro、Bugly、NativeGallery 等第三方库。
- 游密/IM、平台 SDK 等外部服务接入。

### 1.3 工程规模快照
- C# 文件数量、程序集数量、主要业务模块文件量。
- 场景数量和场景用途初判。
- UI、配置、协议、资源目录规模。
- 已生成代码、第三方代码、业务代码的边界。

## 2. 架构总览

### 2.1 启动与生命周期架构
- `GameLaunch` 启动阶段：帧率、SDK 初始化、语音 SDK、AssetBundle 初始化、资源更新、图集回调、进入游戏主流程。
- `GameMain` 主流程：热更 DLL、Manager 启动、配置加载、玩法数据初始化、登录场景加载。
- Update/FixedUpdate/LateUpdate 的分发对象和职责。
- 应用暂停/恢复/焦点变化的处理链路。

### 2.2 分层结构
- 基础层：Core / GameCore / GameCommon / Manager。
- 数据层：Config / DataManager / Proto。
- 通信层：Net。
- 业务层：GamePlay / Battle / Character / Scene / Timeline。
- 表现层：UI / 资源 / 场景 / 特效 / Spine。
- 接入层：SDK / Bugly / YouMe IM / NativeGallery 等。

### 2.3 程序集与依赖关系
- 汇总 `.asmdef` 依赖，绘制模块依赖图。
- 识别强耦合模块、反向依赖、公共层被业务污染的风险。
- 标注可热更程序集与非热更程序集边界。

### 2.4 资源与热更新架构
- AssetBundle 初始化、平台选择、更新模式判断、下载进度反馈。
- HybridCLR 元数据加载、DLL 下载和启动。
- Addressables/AssetBundle/StreamingAssets/Resource 的实际使用边界。
- 资源命名、图集、场景、Prefab、音乐和配置资源的组织方式。

## 3. 功能模块总结

### 3.1 核心玩法
- 舞蹈/节奏/动作相关逻辑。
- Battle 模块职责：战斗或玩法局内逻辑、评分、结算、同步等。
- Character 模块职责：角色、形象、动作、装扮、骨骼/动画驱动。
- Timeline 模块职责：歌曲、节拍、动作时间线或演出编排。

### 3.2 大厅与社交
- 登录、房间、大厅、婚礼、拍照、幽灵房等场景入口。
- 房间系统、聊天/语音、玩家互动、关系链、同伴/伴侣等功能初判。
- 对应 UI、协议、数据结构和场景资源。

### 3.3 UI 系统
- UI 框架与窗口管理方式。
- UI 模块规模、命名规范、Base 类、资源加载和生命周期。
- 主要界面清单：登录、主界面、房间、背包、商城、角色、设置、弹窗等。
- UI 完成度和重复/废弃界面风险。

### 3.4 网络与协议
- NetMgr 启动、网络循环、消息派发机制。
- Proto 目录和协议生成方式。
- 登录、房间、角色、背包、商城、社交等协议覆盖度。
- 客户端与服务端联调成熟度迹象。

### 3.5 配置与数据
- Luban 配置体系和生成代码。
- ConfigMgr 初始化流程。
- 配置表覆盖范围：音乐、角色、道具、UI、房间、活动等。
- DataMgr 管理的运行时数据域和持久化策略。

### 3.6 SDK 与平台能力
- 平台 SDK 初始化和生命周期。
- 语音/IM、Bugly、相册、权限、支付/登录/实名等可能接入点。
- Android/iOS 构建配置和平台差异。

## 4. 现阶段开发进度分析

### 4.1 进度判断方法
- 源码完成度：模块是否有入口、数据结构、协议、UI、资源绑定。
- 功能闭环：是否从配置、网络、数据、UI、场景到运行流程形成闭环。
- 资源完备度：场景、Prefab、图集、音乐、动画、Spine、特效是否具备可运行内容。
- 工具链完备度：配置生成、热更构建、资源打包、编辑器工具是否可用。
- 近期活跃度：按文件修改时间和模块变更密度判断开发重心。
- Bug 修复期可信度：区分“功能已完成但修 Bug”和“仍在补核心链路”的差异。
- 重构后稳定度：重点检查底层模块是否仍存在过渡代码、重复框架、废弃入口、兼容胶水和绕路实现。

### 4.2 里程碑状态初判
- 原型期：核心玩法和基础 UI 是否可进入。
- Alpha：主要系统是否有代码骨架和测试资源。
- Beta：多数功能是否形成可测闭环。
- 上线前：性能、稳定性、支付合规、热更、安全、包体、灰度、监控是否齐备。
- 首次技术测试前：登录、更新、进服、核心玩法、房间/社交、异常恢复、崩溃收集、日志和问题定位是否具备最小闭环。

### 4.3 模块进度矩阵
- 维度：设计完整度、代码完整度、资源完整度、配置完整度、协议完整度、联调状态、测试风险。
- 评级：已形成闭环 / 基本可测 / 半成品 / 仅有骨架 / 未见实现。
- 覆盖模块：启动热更、登录、角色、房间、大厅、舞蹈玩法、战斗/结算、UI、商城/经济、社交、聊天语音、拍照、婚礼、背包、配置、网络、SDK。

### 4.4 交付风险
- 技术风险：热更链路、资源更新、程序集依赖、网络同步、配置生成、SDK 稳定性。
- 内容风险：歌曲/谱面/动作/角色/场景资源量与质量。
- 体验风险：首包和热更耗时、登录流程、UI 响应、低端机性能。
- 项目风险：第三方库、编码规范、缺少自动化测试、构建环境复现难度。

### 4.5 首次技术测试 readiness
- 测试目标是否清晰：技术验证、留存验证、玩法验证、服务器压测、兼容性验证分别覆盖哪些内容。
- 测试范围是否收敛：哪些系统必须开放，哪些系统可以关闭或弱化。
- 可测闭环：下载/安装、启动、热更、登录、创角/进服、进入大厅、进入核心玩法、结算、退出重进。
- 稳定性底线：崩溃率、卡死、资源丢失、协议断线、配置缺失、异常弹窗、黑屏白屏。
- 定位能力：客户端日志、服务端日志、Bugly/崩溃上报、账号维度追踪、版本号和资源号识别。
- 测试后复盘材料：问题分级、复现路径、责任模块、修复成本、是否影响下一轮测试。

### 4.6 客观评价口径
- 不是只看“代码量/界面数量”，而是看核心链路是否稳定跑通。
- 不是只看“当前能跑”，还要看重启、断网、弱网、热更失败、配置异常、低端机等边界情况。
- 不是只看“研发说完成”，而是按模块证据表给出已闭环/可测/半成品/骨架/缺失。
- 对制作人输出一句话评级：进度、质量、风险、首测建议。

## 5. 工程质量与可维护性

### 5.1 代码结构质量
- Manager 模式的集中度和职责边界。
- 单例、静态类、事件广播、全局状态使用情况。
- 命名规范、目录规范、模块内聚性。
- 业务逻辑和表现逻辑是否混杂。
- 历史改动痕迹：废弃代码、重复系统、兼容层、临时分支、注释掉的逻辑、未删除旧框架。
- 基础模块稳定性：资源、网络、配置、UI、场景、数据管理是否存在反复改动后的边界不清。

### 5.2 可测试性
- Unity Test Framework 使用情况。
- 是否存在 EditMode/PlayMode 测试。
- 关键链路是否可脚本化验证：启动、配置加载、协议解析、资源加载、UI 打开。

### 5.3 性能与包体
- 资源目录和 StreamingAssets/AssetBundle 包体风险。
- UI 数量、贴图、Spine、特效、音频、场景大小。
- 内存、GC、异步加载、对象池、图集策略。

### 5.4 安全与合规
- 客户端密钥、Keystore、SDK 配置、日志敏感信息。
- 支付、登录、实名、防沉迷、隐私权限。
- 热更安全、资源校验、协议校验。

## 9. 后续填充顺序建议

1. 先完成《首测 readiness 快评》和《启动/热更/登录/进服/核心玩法》最小闭环检查，因为它们直接决定下个月能否测试。
2. 再完成《项目总览》和《启动/生命周期架构》，建立全局地图。
3. 接着做《功能模块总结》和《模块进度矩阵》，形成制作人可用的进度判断。
4. 然后重点补《热更/资源/配置/网络/UI/场景》六条关键基础链路，识别历史重构后的稳定性风险。
5. 最后补《工程质量》《问题清单》《后续行动建议》，用于首测前排期和团队对齐。

# 资源与配置一致性扫描

生成时间：2026-06-29  
分析目标：检查首测前最容易造成黑屏、卡 loading、UI 打不开、配置报错、真包启动失败的资源与配置一致性问题。

## 1. 总体结论

本轮扫描重点检查了 UIConfig 与 Prefab、配置声明与导出 JSON、场景与 BuildSettings、热更 DLL/AOT、StreamingAssets 与资源体量。结论如下：

> 配置 JSON 和场景配置整体比较健康；UIConfig 存在 22 个 PrefabPath 在常规资源目录中找不到，其中大多集中在活动、VIP、CP 商城、礼物、战斗测试等外围功能；热更 DLL/AOT 不能仅从当前目录确认完整，必须在正式出包流程中做清单校验。

对首测影响最大的不是配置 JSON，而是：

1. **UIConfig 配置了不存在或路径漂移的 UI Prefab**。
2. **热更 DLL/AOT 元数据当前目录状态不能证明真包完整**。
3. **资源体量大，动画/场景/角色/音频是包体和加载压力核心**。
4. **StreamingAssets 使用大量哈希 `.a` 包，必须依赖 manifest/asset map 校验，不适合人工肉眼确认**。

### 3.1 影响判断

这 22 个问题大多集中在前面建议首测隐藏的外围模块：活动、首充、礼包、抽卡、VIP、CP 商城、礼物、战斗测试。它们不一定会影响登录、频道、广场、普通房间、基础战斗这条主链路。

但风险在于：

- 如果活动配置、任务跳转、主界面入口或 GM 入口打开这些 UI，会运行时加载失败。
- `UIActivityCheckIn` 是路径漂移，不是资源完全不存在；如果签到活动开放，会直接受影响。
- VIP、首充、礼包、抽卡这类商业化入口如果误开，会造成很差的首测观感。

### 3.2 首测处理建议

| 处理项 | 优先级 | 建议 |
|---|---|---|
| 修正 `UIActivityCheckIn` 路径 | P0/P1 | 如果开放签到，必须修正；不开放则仍建议清理 |
| 活动/首充/礼包/抽卡 UI 缺失 | P0 | 首测隐藏相关入口和服务端活动配置 |
| VIP UI 缺失 | P0 | 首测隐藏 VIP 入口 |
| CP 商城缺失 | P1 | 首测不要开放 CP 商城 |
| 礼物相关 UI 缺失 | P1 | 房间礼物、好友赠礼需限制 |
| `UIBattleTest` 缺失 | P2 | 测试 UI 残留，可清理配置 |

## 4. 配置表与 JSON 一致性

扫描结论：

- `ConfigManager._configNames` 声明 69 项。
- 去重后 68 项。
- `Assets/AssetsPackage/Config/cfg` 下 JSON 为 68 项。
- 缺失：0。
- 多余：0。
- 重复声明：`data_tbdancestepbylevelfree` 出现 2 次。

这说明配置导出总体是健康的，是本轮扫描里最稳的一块。

### 4.1 重复项

| 配置名 | 次数 | 影响 |
|---|---:|---|
| `data_tbdancestepbylevelfree` | 2 | 通常不阻断运行，但会重复请求/重复赋值，建议清理 |

### 4.2 研发建议

首测前建议增加自动检查：

1. `_configNames` 中不允许重复。
2. `_configNames` 每一项必须有对应 JSON。
3. JSON 中不允许有 `_configNames` 未声明的文件。
4. 每个 JSON 能被 Luban/Tables 正常解析。
5. 配置引用的 UIFlag、SceneId、ItemId、MusicId、PrefabPath 要继续做二级引用校验。

特别提醒：

当前只是“配置文件存在性”通过，不代表配置内容引用的资源都存在。比如活动配置可能引用缺失 UI、缺失图标、缺失奖励或未开放跳转。

扫描结论：

场景列表：

| 场景 | 用途判断 |

## 6. 热更 DLL 与 AOT 元数据

源码中 `LoadDll` 固定加载 13 个热更 DLL：

| DLL |
|---|
| `Config.dll` |
| `GameCore.dll` |
| `GameCommon.dll` |
| `Proto.dll` |
| `Manager.dll` |
| `DataManager.dll` |
| `Character.dll` |
| `Battle.dll` |
| `GamePlay.dll` |
| `Net.dll` |
| `Scene.dll` |
| `UI.dll` |
| `Timeline.dll` |

当前扫描结果：

- `Assets/StreamingAssets` 下没有直接以 `.dll` 文件名存在的热更 DLL。
- `StreamingAssets/pkg` 下存在大量哈希 `.a` 文件，说明资源可能已被打包/重命名，不能用文件名直接判断缺失。
- `HybridCLRData/HotUpdateDlls/Android` 下能找到 `Config/GameCore/GameCommon/Proto/Manager/DataManager/Character` 等 DLL。
- 但在该目录直接未找到 `Battle/GamePlay/Net/Scene/UI/Timeline`。
- AOT 元数据固定列表 `System.dll`、`System.Core.dll`、`mscorlib.dll`、`Framework.dll` 在当前常规目录下未直接扫到。

### 6.1 风险解释

这里不能简单判断“真包缺 DLL”，因为正式包可能通过 AssetBundle 构建流程把 DLL 打进 `pkg/*.a`，并通过 `AssetBundleManager.RequestAssetFileAsync(asset)` 按 asset 名加载。

但对首测来说，这仍然是 P0 风险，因为：

- 源码依赖固定 DLL 名称。
- 真包里 DLL 是否进入 AssetBundle，必须通过 asset map/manifest 验证。
- AOT 元数据如果缺失，HybridCLR 真机运行可能出现泛型、反射、AOT 补充元数据相关崩溃。
- 当前目录状态不足以证明正式 Android 包完整。

### 6.2 首测前必须补的校验

研发需输出以下清单：

1. 本次首测包实际包含的热更 DLL 清单。
2. 每个 DLL 对应 AssetBundle asset 名。
3. `AssetBundleManager.RequestAssetFileAsync("UI.dll")` 等 13 项在真包中是否能成功加载。
4. AOT 元数据文件清单。
5. HybridCLR 宏定义和 Android IL2CPP 设置截图或 CI 日志。
6. 真机启动日志，逐项打印 DLL 和 AOT 元数据加载成功。

## 9. 建议研发增加的自动化检查

首测前建议至少做 5 个自动化检查：

1. **UIConfig-Prefab 检查**  
   遍历 `UIConfig._uiMap` 中所有 `PrefabPath`，确认资源存在于出包资源清单。

2. **ConfigManager-JSON 检查**  
   `_configNames` 去重、查缺、查多余、Luban 解析校验。

3. **Scene-BuildSettings 检查**  
   `SceneConfig`、`SceneManager`、BuildSettings 三者一致。

4. **HotUpdate DLL/AOT 检查**  
   对 `LoadDll` 固定列表逐项验证 AssetBundle 加载成功。

5. **首测入口检查**  
   对隐藏功能的 UIFlag、主界面按钮、广场触发点、活动配置、任务跳转做一致性检查。

建议这些检查进入每日构建，而不是人工出包前临时看一眼。

## 10. 制作人关注点

建议你在首测前直接追问研发：

1. UIConfig 这 22 个缺失/路径不匹配 UI，哪些会在首测开放？
2. `UIActivityCheckIn` 路径是否要修？签到活动是否开放？
3. VIP、首充、礼包、抽卡、CP 商城是否全部隐藏？
4. 正式首测 Android 包能否输出 13 个热更 DLL 的加载成功日志？
5. HybridCLR AOT 元数据是否进入包体或热更资源？日志在哪里看？
6. 首测首包/热更包体积是多少？是否会让玩家首次下载压力过大？
7. 不开放的拍照、婚礼、心动房资源是否仍在首包里？
8. 每日构建是否自动跑 UI/配置/场景/热更清单检查？

## 11. 客观评价

资源与配置层面的整体评价：

> 配置 JSON 和场景配置相对健康；UI 配置存在外围模块缺失/路径漂移；热更与 AOT 需要正式包级别验证；资源体量较大，首测要控制开放范围和包体策略。

这份扫描进一步支持前两份报告的建议：首测应收敛到主链路，不要贸然开放活动、VIP、首充、礼包、CP 商城、拍照、心动房、婚礼等外围功能。  
否则玩家遇到的可能不是“玩法不好玩”，而是“点入口没反应、UI 打不开、loading 卡住、资源加载失败”。

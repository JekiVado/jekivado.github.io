# 功能模块完成度评估

生成时间：2026-06-29  
分析目标：按游戏功能模块评估当前完成度、首测开放建议、主要风险和验收重点，为下月首次技术测试确定功能边界。

## 1. 总体结论

当前项目功能面已经铺得比较宽，不是“只有主链路”的状态。登录、频道、广场、房间、战斗、结算、聊天、背包、好友、商城、任务、活动、拍照、婚礼、心动房、家族、排行、邮件、成就、服装兑换等都有不同程度的代码和 UI 支撑。

但首测不建议按“已有代码就开放”的原则推进。当前更合理的策略是：

> 首测以验证主链路稳定为第一目标：启动、登录、进频道、进广场、进房间、打一局、结算、基础聊天、断线处理、崩溃采集。  
> 外围系统应分级开放，尤其是家族、拍照、心动房、婚礼、复杂活动、服装兑换、支付分享，不宜一次性全部放开。

功能模块按首测开放建议分为四档：

| 档位 | 建议 | 模块 |
|---|---|---|
| A | 首测必须开放并重点验收 | 启动热更、登录创角、频道、广场、普通房间、基础战斗、结算 |
| B | 可以开放，但需要控制范围 | 聊天、背包换装、好友、商城浏览/非真实支付、邮件、任务/成就、排行榜等级榜、设置 |
| C | 建议首测隐藏或灰度 | 家族、拍照馆、心动房、婚礼、复杂活动抽奖、服装兑换、蓝图、VIP/礼包/首充 |
| D | 必须研发确认后再决定 | 支付、分享、渠道 SDK、语音权限、相册保存、账号绑定、问卷/外链 |

## 2. 评估方法

评估不是只看代码行数，而是综合以下信号：

- 是否有 UI 入口。
- 是否有 UI Prefab 和 UIConfig 配置。
- 是否有 DataManager 状态管理。
- 是否有 Net ProtoExtensions 协议收发。
- 是否在场景或主界面中存在实际入口。
- 是否存在 TODO、临时方案、暂未开放、协议初始化注释。
- 是否依赖外部 SDK、原生权限、服务端配置、活动时间或支付环境。
- 是否位于首测主链路。

评分说明：

| 分数 | 含义 |
|---|---|
| 5 | 可作为首测核心功能，仍需真机/真服验收 |
| 4 | 基本完整，可开放但需要明确边界 |
| 3 | 有完整雏形，但风险偏高，建议灰度或限制入口 |
| 2 | 代码不少但成熟度不足，首测不建议开放 |
| 1 | 仅有残留、入口或配置，需研发确认 |

## 4. A 档：首测必须开放

### 4.1 启动、热更、资源

完成度：4/5  
首测建议：必须开放，作为所有测试前置条件。

证据：

- `Assets/Scripts/GameLaunch.cs` 负责 SDK 初始化、AssetBundle 初始化、更新检查和进入 `GameMain`。
- `Assets/Scripts/GameMain.cs` 串联 `InitDll`、`InitMgr`、`StartGame`。
- `Assets/Scripts/GameCore/LoadDll.cs` 固定加载 13 个热更 DLL 和 HybridCLR AOT 元数据。

主要风险：

- 真机包 DLL/AOT 缺失。
- AssetBundle manifest 与实际资源不一致。
- 编辑器路径与真包路径差异。
- Bugly 当前疑似未启用。

首测验收：

- 首装启动、二次启动、清缓存启动。
- 有更新、无更新、弱网更新、中断恢复。
- 启动日志能看到 DLL/AOT/配置/场景加载关键节点。

### 4.2 登录、创角、频道

完成度：4/5  
首测建议：必须开放。

证据：

- `LoginScene` 打开 `UILogin` 并在场景加载完成后广播网络连接。
- `LoginProtoExtensions` 处理账号登录、角色登录、创角、踢号等。
- `UIChannel`、`ChannelDataManager`、`ChannelProtoExtensions` 支撑频道列表和加入频道。

风险：

- `LoginProtoExtensions` 中有“临时方案，防止服务器发多次登录，CB5后再看”。
- 频道加入请求发出时，客户端先广播 UI 加入结果，`UIChannel` 会立即打开 loading 并关闭自身；失败或超时可能卡住。
- 登录后同时触发数据初始化、UI 打开、YouMe 登录，任意一环失败都可能表现为登录卡住。

首测验收：

- 新账号创角进入频道。
- 老账号登录进入频道。
- 重复点击登录。
- 服务端重复回包。
- 频道加入失败、超时、断线时有回退。

### 4.3 广场主界面

完成度：4/5  
首测建议：必须开放。

证据：

- `SquareScene.OnPreLoadScene()` 打开 `UIMain`。
- `SquareScene` 触发点可打开房间、家族、排行、商城和拍照房。
- `UIMain` 提供背包、好友、设置、邮件、成就、排行、礼物、拍照、客服等入口。

风险：

- 广场承担过多外围入口，首测如果入口不做屏蔽，玩家会很快进入未稳定模块。
- `MainIconDataManager` 基于配置动态展示主界面活动入口，运营配置错误会影响首测体验。

首测建议：

- 广场必须开放。
- 但首测版应明确隐藏或置灰 C/D 档入口。
- 广场触发点也要按首测范围配置，尤其家族、拍照、复杂活动。

### 4.4 普通房间

完成度：4/5  
首测建议：必须开放。

证据：

- `RoomDataManager` 支持房间列表、创建、进入、密码房、邀请。
- `RoomProtoExtensions` 注册创建、加入、离开、修改、准备、邀请、座位、阵营等协议。
- `UIRoom` 支持准备、开始、换背景、修改房间、邀请、聊天、礼物、好友、背包、音乐选择等。

风险：

- 房间功能点多，首测应优先验证普通房间，不建议一开始开放所有房间模式。
- 房主权限、座位、准备状态、房间音乐、离开房间、断线后的状态恢复都容易出问题。

首测验收：

- 创建普通房间。
- 加入他人房间。
- 准备/取消准备。
- 房主开始。
- 离开房间再进入。
- 房主离开、普通玩家离开。

### 4.5 基础战斗和结算

完成度：4/5  
首测建议：必须开放。

证据：

- `BattleProtoExtensions` 注册进入、开始、提交、结算等战斗消息。
- `BasketballScene` / `BattleScene` 打开 `UIBattle`、`UIBattleDGroup`、`UIBattleSingle`。
- `UIBattle` 包含音乐、分数、排名、观战、方向键、空格、视频、聊天缩放等组件。

风险：

- 战斗输入、服务端时间、提交判定、结算奖励、断线重连都需要真服验证。
- 战斗 UI 支持多模式，首测应限制到核心模式。
- `UIArrowList` 中存在测试数据片段，需确认不会进入正式路径。

首测验收：

- 单人/多人进入同一局。
- 开始倒计时。
- 正常输入。
- 战斗提交。
- 战斗结算。
- 返回房间或广场。
- 战斗中断网、切后台、重连。

## 5. B 档：可开放但控制范围

### 5.1 聊天和语音

完成度：3/5  
首测建议：基础文字聊天可开放，语音能力需专项验收。

证据：

- `ChatDataManager`、`PanelChatNewest`、`UIChatScalingPanel`、`YIMSDK`/`IMAPI` 代码存在。
- `GameLaunch` 初始化 YouMe SDK，并设置音频播放监听。
- 登录成功后会广播 YouMe 登录。
- 频道/房间会加入或离开对应聊天频道。

风险：

- 依赖第三方 SDK、麦克风权限、录音、语音播放、频道状态。
- 家族聊天相关代码里有 TODO，且家族协议当前未启用。
- 频道/房间加入失败可能连带聊天状态错乱。

首测建议：

- 开放频道文字聊天、房间文字聊天。
- 语音聊天、私聊、家族频道、语音转文字视测试准备程度决定。

### 5.2 背包、换装、个人信息

完成度：3.5/5  
首测建议：可开放基础查看和换装，限制复杂赠送/套装/特殊道具。

证据：

- `UIBag`、`ItemDataManager`、`PlayerProtoExtensions`、`UIInformation*` 均存在。
- 主界面和房间 UI 都提供背包入口。
- 道具预览、头像、称号、徽章、戒指、情侣信息等入口存在。

风险：

- 背包和外观涉及角色表现、场景玩家同步、商城购买、奖励发放。
- 道具类型多，配置错误容易导致预览、穿戴或同步异常。

首测建议：

- 开放基础背包、基础换装。
- 高价值/付费/限时/情侣/赠送类道具先收敛。

### 5.3 好友

完成度：3.5/5  
首测建议：可开放基础好友功能，谨慎开放深层关系。

证据：

- `FriendDataManager`、`FriendProtoExtensions`、`UIFriendView`、`UIFriendPlayerCard` 等完整。
- 任务系统、商城赠送、房间邀请、聊天都能进入好友相关 UI。

风险：

- 好友系统和聊天、赠送、房间邀请、情侣关系耦合。
- 深层社交链路容易产生服务端状态边界问题。

首测建议：

- 开放好友列表、搜索、加好友、基础邀请。
- 情侣关系、赠送、跨服邀请、黑名单等可以后置。

### 5.4 商城

完成度：3/5  
首测建议：可以开放浏览和非真实支付路径，真实支付谨慎。

证据：

- `ShopManager`、`ShopProtoExtensions`、`UIShop`、`UICpShop`、`UIPersonalShop` 存在。
- `JWGameSDK` 包含 Android 登录、支付、分享接口。

风险：

- 支付依赖 Android 原生 SDK 和支付环境。
- 商品、货币、购物车、赠送、性别限制、时效道具都依赖配置。
- 首测如果开放真实支付，客服和退款风险上升。

首测建议：

- 开放商城浏览。
- 购买走测试货币或关闭真实支付。
- 如果必须测支付，单独小范围专项测试，不纳入普通玩家首测主流程。

### 5.5 邮件、任务、成就、设置

完成度：3-4/5  
首测建议：可开放，但任务/成就奖励链路需控制。

证据：

- `MailDataManager`、`MailProtoExtensions`、`UIEMail` 存在。
- `TaskDataManager`、`TaskProtoExtensions`、`UITaskCenter` 存在，且任务跳转会打开背包、好友、聊天、房间等功能。
- `AchievementsDataManager`、`UIAchievements` 存在。
- `UISetting`、`SettingDataManager`、`SettingManager` 存在。

风险：

- 任务会把玩家引到多个外围系统，可能突破首测功能边界。
- 奖励发放涉及道具、邮件、背包、红点。
- 设置风险较低，但音量、权限、自定义设置仍需真机验证。

首测建议：

- 邮件可用于发放测试补偿和奖励。
- 任务/成就只开放少量基础任务，不要引导玩家进入 C 档系统。
- 设置开放基础音量、画质、账号相关前需确认渠道行为。

## 6. C 档：建议隐藏或专项灰度

### 6.1 家族

完成度：2/5  
首测建议：隐藏。

证据：

- `FamilyDataManager`、`UIFamily`、`FamilyProtoExtensions` 都存在。
- `FamilyProtoExtensions` 注册 19 个服务端消息和 21 个事件请求。
- 但 `NetworkManager` 中 `FamilyProtoExtensions.Init()` 和 Dispose 当前被注释。
- 广场触发点仍会打开 `UIFamily` 并请求家族列表。

判断：

这是典型“代码看起来不少，但运行时开关未打开”的模块。首测如果入口没屏蔽，玩家点到家族可能出现 UI 打开但协议无响应。

建议：

- 首测隐藏家族入口。
- 若必须开放，先恢复并验证协议初始化、服务端接口、红点、聊天家族频道。

### 6.2 拍照馆

完成度：2.5/5  
首测建议：隐藏或专项灰度。

证据：

- `PhotoStudio` UI 体量较大，UI 代码约 6140 行。
- 广场和主界面均可创建拍照房。
- `PhotoStudioRegistry` 注册拍照相关协议。
- 代码中存在 TODO，如邀请反馈、统计回包、事件埋点“函数缺失，等补全”。
- 依赖相册保存、分享、拍照 UI、多人房间同步。

风险：

- 功能链路长：创建拍照房、邀请、换装、站位、拍摄、保存、分享。
- 依赖 NativeGallery、Android 权限、渠道分享 SDK。
- TODO 明确存在，首测容易暴露未收口边界。

建议：

- 普通首测隐藏拍照入口。
- 可安排研发/QA 内部专项灰度，目标是验证相机、保存、分享、邀请。

### 6.3 心动房/心动抽奖

完成度：2.5/5  
首测建议：隐藏或专项灰度。

证据：

- 存在 `HeartbeatMateRoom` 场景、`UIHeartbeatMate`、`HeatRoomList`、`HeartDraw`、`SoulmateManager`。
- `SoulmateProtoExtensions` 文件存在，但协议统计中 Register/Events/Sends 为 0。
- 心动房 UI 和场景逻辑与礼物、好友、房间邀请、抽奖等交织。

风险：

- 社交玩法状态复杂。
- 礼物、抽奖、房间、好友关系同时参与。
- 协议扩展是否完整需研发确认。

建议：

- 首测不作为公开功能。
- 如果产品必须验证社交卖点，建议只做小规模定向测试。

### 6.4 婚礼

完成度：2.5/5  
首测建议：隐藏。

证据：

- 存在 `WeddingRoomScene`、`WeddingScene`、`WeddingManager`、`TimelineManager`、`UIWedding*`。
- 婚礼场景会打开房间 UI、战斗 UI、婚礼目标/任务/照片/结果等。

风险：

- 依赖 Timeline、多玩家同步、战斗、房间、婚礼任务和结果。
- 不是首测主链路，开放会显著放大测试面。

建议：

- 首测隐藏。
- 后续做婚礼专项测试。

### 6.5 活动、服装兑换、蓝图、VIP、礼包

完成度：1.5-2.5/5  
首测建议：隐藏或只开单一简单活动。

证据：

- `ActivityDataManager` 会从配置表初始化活动时间、展示时间、奖励、抽奖券、保底等。
- `ActivityProtoExtensions`、`TaskProtoExtensions`、`ClothingProtoExtensions` 均有多条协议。
- UIConfig 中存在大量活动、抽奖、礼包、首充、服装兑换、蓝图相关 UI。

风险：

- 极度依赖运营配置、时间配置、奖励配置、服务端活动状态。
- 抽奖、兑换、保底、首充、礼包都涉及付费或经济系统。
- UIConfig 曾初步发现部分配置路径需复查，活动类尤其容易配置漂移。

建议：

- 首测只开放最简单的签到或公告类活动。
- 抽奖、首充、礼包、VIP、蓝图、服装兑换不建议面向普通首测玩家开放。

## 7. D 档：需要研发确认

| 模块 | 为什么需要确认 | 建议 |
|---|---|---|
| 支付 | Android SDK 接口存在，但需沙盒/渠道/订单/回调全链路 | 首测默认关闭真实支付 |
| 分享 | 抖音/微信/朋友圈等提示存在，依赖渠道 SDK 和权限 | 只做内部专项 |
| 相册保存 | 拍照保存依赖 NativeGallery 和系统权限 | 随拍照专项验证 |
| 语音权限 | YouMe 录音/播放/语音消息依赖麦克风权限 | 首测前做权限矩阵 |
| 客服/问卷 | 可能依赖外链、活动配置或 WebView | 确认 URL、隐私、奖励发放 |
| 账号绑定 | UIConfig 中有账号绑定相关入口 | 渠道账号体系确认后再开 |

## 9. 首测入口治理建议

首测前请研发输出一份“入口开关表”，至少包含：

| 入口 | 当前来源 | 首测状态 |
|---|---|---|
| 主界面菜单入口 | `UIMain` 按钮 | 按开放范围保留/隐藏 |
| 广场触发点 | `SquareScene` Trigger1-Trigger5 | 家族/拍照等需屏蔽 |
| 活动入口 | `MainIconDataManager` 配置 | 首测只保留确认活动 |
| 任务跳转 | `TaskDataManager` | 不跳转到隐藏功能 |
| UIConfig 入口 | `UIConfig` | 隐藏功能不应被误打开 |
| 服务端活动/功能开关 | 服务端配置 | 与客户端入口一致 |

重点不是“把按钮隐藏”这么简单，还要确保：

- 任务不会跳转到隐藏功能。
- 活动配置不会动态挂出隐藏入口。
- 红点不会提示隐藏功能。
- 服务端不会推送隐藏功能的奖励或跳转。
- GM/测试入口不会泄露给普通测试玩家。

## 10. 制作人关注清单

建议你在首测排期会上直接问研发以下问题：

1. 哪些功能首测确定开放？哪些隐藏？有没有入口开关表？
2. 家族协议为什么当前初始化是注释？家族入口是否会屏蔽？
3. 拍照馆是否要进首测？如果进，TODO 和相册/分享权限是否已验收？
4. 心动房、婚礼、服装兑换、活动抽奖是否有服务端完整环境？
5. 支付是否完全关闭？商城是否只测浏览和测试货币？
6. 任务和活动是否会把玩家引到未开放模块？
7. 首测包里 Bugly/日志/关键事件埋点是否启用？
8. 如果玩家卡 loading、频道失败、战斗中断线，客户端如何回退？

## 11. 客观评价

以当前源码看，项目功能完成度的真实状态应描述为：

> 主链路基本具备首测条件；外围功能代码覆盖较广，但成熟度不均。  
> 当前阶段最重要的不是继续证明“功能都有”，而是收敛首测范围，把主链路打稳，把高风险外围功能延后或灰度。

如果强行开放过多外围模块，首测反馈很可能被以下问题淹没：

- 家族入口无响应。
- 拍照保存/分享权限失败。
- 心动房/婚礼多人状态错乱。
- 活动奖励配置错误。
- 支付/礼包/首充环境不一致。
- 任务引导玩家进入未完成系统。

更合理的首测目标是：

> 用较窄但完整的功能范围，验证客户端基础架构、网络稳定性、多人房间、核心战斗、基础社交和日志采集能力。  
> 等主链路稳定后，再分批开放拍照、婚礼、心动房、家族、活动商业化系统。

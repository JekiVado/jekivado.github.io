⚔️ Battle CampRule — 策略模式驱动多样化玩法

这是一个架构层面的重要升级。团队在 Battle 层引入 策略模式 (Strategy Pattern)，将"阵营规则"从 BattleManager 中解耦。以接口 ICampRule 为契约，不同房间模式有不同的光环渲染和命中判定规则。

组件职责说明
ICampRule接口定义定义阵营光环(Common/Perfect/FinishMove)、场景根节点、命中判定规则
CampRuleFactory工厂按 ERoomMode 返回对应规则实例；支持预注册双人模式(Double)，其余默认 Common
CampRuleCommon通用规则默认光环渲染 (Halo_1/2/3)，红/蓝/绿阵营颜色分化
CampRuleDouble双人规则继承 Common，覆盖 HitRule：追踪阵营内每个 ArrowIndex 的 Perfect 数，全员完美命中时广播 MN_BATTLE_ALL_HIT_EFFECT
CampRuleFree自由模式自由式舞步特有规则
CampRuleNormal普通模式常规舞步规则
CampRuleNullObject空对象无规则模式（Null Object Pattern）
技术评价：策略模式的引入使得未来增加新玩法模式（如团队对抗、闯关）只需新增一个 ICampRule 实现并注册到 Factory，不需要修改 BattleManager 核心代码。这是 开闭原则 (OCP) 的良好实践。当前 Factory 中仅预注册了 Double，其余通过 defaultCampRule 兜底，扩展性良好。

💕 心印系统 (HeartbeatMate) — 从 85% 到完整交付

上月心印系统进度持平在 85%。本月迎来大规模代码落地，首次出现完整的 HeartbeatMateManager + Partial 架构。

HeartbeatMateManager 架构 (9 文件, Partial 拆分)

Partial 文件职责关键内容
HeartbeatMateManager核心+配置分析AsyncConfigAnalyseInitialize(): 异步分析 SoulmateGift 配置表，构建多重礼物/双人礼物的数据字典 + 座位位置索引
DrawPartial抽卡心印礼物抽卡、兑换奖励、红色点历史记录
GiftServerPartial送礼服务端礼物队列发送、屏幕滚动消息播放

辅助组件

  • HeartbeatMateSceneUnit — 场景单元管理（位置索引 + 表演单元字典）
  • HeartbeatMatePerformUnit — 表演单元（3D 礼物表现）
  • HeartbeatMateTemplate — 数据模板
  • PlayableLogicEvent — 可播放的逻辑事件
注意：HeartbeatMateManager 不在 GamePlay.Init() 中初始化。它的初始化由 HeartbeatMateRoom 场景控制（HeartbeatMateRoomInitialPartial），这符合"按需加载"的架构原则。当前 Manager 内部使用了 LBS 定位服务（StopLocationService），说明心印房间具备基于地理位置匹配的能力。

🏛️ 广场场景 (SquareScene) — 3D 社交大厅

全新实现的社交中枢场景，玩家可在 6 个建筑物之间点触导航：

建筑物导航
舞厅 (DancingHall) → 房间列表 | 家族大厅 (FymilyHall) → 家族界面

技术亮点

  • 触控交互：SimpleTouch.On_SimpleTap 事件驱动，检测触摸点是否在建筑物触发区
  • 相机动画：CinemachineVirtualCamera + Animation 组件，PoolManager 动态加载相机动画 Clip，进入/退出各 2.2 秒动画
  • 高亮系统：HighlightPlus.HighlightEffect 为 6 个建筑提供边缘高亮（点击时激活，完成动画后关闭）
  • 玩家初始化：复用 ScenePlayerManager.InitAllPlayers()
  • BGM：进入广场后播放默认背景音乐

👔 服装兑换系统 — 完整的抽卡+交易生态

这是本月最大的新增业务系统（17 UI 文件 + 1 DataManager + 多张配置表），实现了"时装箱"玩法的完整闭环。

层级组件说明

技术细节：ClothingExchangeManager 用 MD5 对玩家 UIN 进行签名 (HMAC-MD5 + "&" + timestamp)，调用外部 HTTP API (9you.com) 进行蓝图账号绑定轮询。经理器通过 CancellationTokenSource 控制异步轮询生命周期，300 秒超时自动取消。本地数据采用双向备份策略（Key + Key_Backup）防止数据丢失。

📍 LBS 定位服务 — LocationManager

全新 LocationManager (Singleton) 为项目引入 地理位置能力。这是移动端社交游戏的常见刚需，用于附近玩家匹配、同城房间等 LBS 玩法。

功能实现细节
权限管理Android: Permission.FineLocation + PermissionCallbacks (Granted/Denied/DeniedAndDontAskAgain)；iOS/Editor: 直接返回 Granted
定位服务Input.location.Start()，参数 TimeInterval=120s, DistanceUnit=300m
自动刷新AsyncTask + CancellationTokenSource，每 20 秒刷新一次，通过 EventManager 广播 MN_LOCATION_REFESH
生命周期StartLocationService / StopLocationService (Dispose 时自动 Stop + Cancel)

🚨 遗留问题与风险项

🔴 P0 — GuideManager 不存在 · RedDotManager 仍被注释

GuideManager 在整个 Assets 目录下不存在任何文件。RedDotManager 在 GamePlay.Init() 中仍被注释。新手引导和红点系统持续缺失，这是上线前必须解决的核心问题。

🟡 P1 — BattleScene #if aa 条件编译未清理

BattleScene 仍被 #if aa ... #endif 包裹，BasketballScene 继续作为实际战斗入口。

🟡 P1 — PhotoStudio 埋点 OnEventTracking 仍注释

自 5 月发现后未修复。运营数据埋点缺失将影响上线后的数据分析能力。

🔵 P2 — HeartbeatMateManager 复杂度显著上升

HeartbeatMateManager 拆分为 5 个 partial 文件，Dispose 涉及 10+ 个数据结构的清理、CancellationTokenSource 控制、事件解绑。需要确保所有分支的内存安全。

🔵 P2 — ClothingExchangeManager 外部 HTTP 依赖

蓝图绑定功能依赖外部 9you.com HTTP API + MD5 签名，网络异常/服务不可用时需要有降级方案。

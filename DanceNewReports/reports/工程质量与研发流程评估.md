# 工程质量与研发流程评估

生成时间：2026-06-29  
分析目标：从代码质量、可维护性、测试、构建、配置生产、协议生产、资源生产、日志与监控等角度评估当前工程对首测和后续迭代的支撑能力。

## 1. 总体判断

项目具备较完整的工具链基础，但质量保障体系还没有达到“可放心频繁出测试包”的程度。

一句话评价：

> 工程不是裸奔，有热更、配置、协议、资源、SDK 等工具链；但自动化检查、崩溃采集、弱网兜底、出包清单校验不足，是首测前最需要补的工程能力。

## 2. 工具链现状

已观察到的工具与流程支撑：

| 工具/能力 | 证据 | 判断 |
|---|---|---|
| AssetBundle | `Assets/Editor/AssetBundleMenu.cs`、`GameLaunch` 初始化 AssetBundle | 已接入 |
| HybridCLR | `Packages/manifest.json`、`LoadDll`、`HybridCLRData` | 已接入，但需出包校验 |
| Luban 配置 | `Assets/Editor/LubanExcelToJson.cs`、`LubanConfig` | 已接入 |
| Protobuf | `Assets/Editor/ProtobufMenu.cs`、`Assets/Scripts/Proto` | 已接入 |
| Bugly | `Assets/Editor/BuglyEditor`、插件目录 | 工程存在，但运行时疑似未启用 |
| NativeGallery | 插件和 PostProcessBuild 存在 | 已接入，需权限验证 |
| Unity Test Framework | `Packages/manifest.json` 有依赖 | 依赖存在，但未见测试用例 |
| SVN | 根目录 `.svn` | 使用 SVN 管理 |

## 3. 代码结构质量

### 优点

- 主要业务模块通过 asmdef 拆分，编译边界存在。
- Config、DataManager、Net、UI、Scene、GamePlay 的分层清楚。
- 协议按业务拆成 ProtoExtensions，便于定位。
- UIConfig、SceneConfig、ConfigManager 让资源和配置有统一入口。

### 问题

- UI、Scene、Net、GamePlay 依赖较宽，改动容易牵连。
- 事件总线大量使用，参数和生命周期依赖约定。
- 反射桥接降低编译期保护，真包才容易暴露问题。
- 代码中存在 TODO、临时、暂未开放等信号约 57 处。
- Debug.Log 约 723 处，Debug.LogError 约 168 处，需要区分开发日志和首测日志。

## 4. 可测试性

当前情况：

- Unity Test Framework 包已引入。
- 未扫到 `NUnit`、`UnityTest`、`TestFixture`、`Assert.` 等实际测试用例。
- 首测质量主要依赖人工冒烟和手工 QA。

风险：

- 热更 DLL、UIConfig、配置 JSON、场景 BuildSettings 这类问题适合自动化检查，但目前未见固定测试。
- 每次出包后都可能回归同类问题。
- 修 bug 阶段如果缺少冒烟自动化，会出现“修一个破一个”。

建议优先补最轻量的检查，而不是一开始追求完整自动化：

1. UIConfig 到 Prefab 的存在性检查。
2. ConfigManager 到 JSON 的一致性检查。
3. SceneConfig / SceneManager / BuildSettings 的一致性检查。
4. HotUpdate DLL/AOT 清单检查。
5. 首测入口开关检查。

## 5. 构建与发布风险

当前风险点：

- Android 使用 IL2CPP。
- ProjectSettings 中只看到 iPhone 配置了 `UNITY_HYBRIDCLR`，Android 侧是否由构建流程注入需确认。
- `StreamingAssets` 使用哈希包，人工无法直接确认 DLL/AOT 是否完整。
- 热更和资源更新对首测成败影响极高。

建议：

- 每个首测候选包必须输出构建记录。
- 构建记录包含 Unity 版本、包版本、资源版本、服务器环境、宏定义、热更 DLL 清单、AOT 清单。
- 每个包都保留可追溯的 asset map/manifest。

## 6. 配置与协议生产流程

### 配置

优点：

- Luban 配置流程存在。
- 当前 ConfigManager 声明与导出 JSON 一致，68 个唯一配置无缺失。

问题：

- `_configNames` 中有重复项。
- 文件存在不等于内容引用正确。
- 活动、任务、奖励、UIFlag、资源路径仍需二级引用校验。

### 协议

优点：

- Protobuf 菜单和 Proto 生成代码存在。
- Net ProtoExtensions 按业务拆分。

问题：

- 家族协议文件完整但初始化被注释，说明“生成了协议”不等于“运行时开放”。
- 需要明确每个首测开放系统的协议注册状态。

## 7. 日志与监控

当前问题：

- Bugly 插件存在，但 `GameLaunch.Awake()` 中 `BuglyInit` 被注释。
- Debug 日志数量多，但缺少面向首测的关键链路日志规范。
- 如果首测玩家卡 loading，需要能判断卡在热更、DLL、配置、登录、频道、场景、资源加载还是服务端回包。

建议增加关键埋点/日志：

- 启动开始/结束。
- AssetBundle 初始化开始/结束。
- 热更检查开始/结束。
- 每个 DLL/AOT 加载结果。
- 配置加载完成。
- 连接服务器成功/失败。
- 登录账号/登录角色成功/失败。
- 频道请求发出/回包/失败/超时。
- 场景请求发出/回包/加载完成。
- 房间创建/加入/准备/开始。
- 战斗进入/开始/结算。

## 8. 包体与性能风险

当前资源体量较大：

- `StreamingAssets` 约 3.56 GB。
- `AssetsPackage/Animation` 约 5.84 GB。
- `AssetsPackage/Scene` 约 2.00 GB。
- `AssetsPackage/Role` 约 1.28 GB。
- `AssetsPackage/Sound` 约 1.11 GB。

风险：

- 首包体积和更新体积过大。
- 低端机加载动画/场景/角色资源可能出现卡顿。
- 弱网下热更失败率高。

建议：

- 首测首包和热更包体积必须提前测量。
- 不开放的拍照、婚礼、心动房、活动资源尽量不进入强制下载。
- 首测机型覆盖低端 Android。

## 9. 安全与合规

需要确认：

- Android 权限：麦克风、相册、存储、网络。
- 语音 SDK 隐私合规。
- 相册保存与分享授权。
- 支付入口是否关闭。
- 用户日志是否包含敏感信息。
- 首测包是否连接独立测试环境。

当前最紧急的是支付与分享：

- `JWGameSDK` 包含支付和分享接口。
- 首测如果不是付费测试，真实支付应关闭。
- 拍照分享如果开放，需要单独做权限和平台审核验证。

## 10. 研发流程建议

首测前建议建立最小流程：

| 流程 | 最低要求 |
|---|---|
| 每日构建 | 每天至少一个 Android 候选包 |
| 冒烟记录 | 启动到完成一局的结果可追踪 |
| P0 问题清单 | 每日更新 owner、状态、预计关闭时间 |
| 出包清单 | 包版本、资源版本、服务器、DLL/AOT、配置、场景 |
| 入口开关表 | 首测开放/隐藏功能明确 |
| 崩溃看板 | 真包崩溃可查 |
| 弱网测试 | 登录、频道、房间、战斗四类断线场景 |

## 11. 客观评价

工程质量当前可以支撑一次“有边界的首次技术测试”，但还不适合无边界放量。  
如果补齐出包清单、崩溃采集、主链路日志、入口治理和基础自动检查，首测成功率会明显提高。

优先级排序：

1. 崩溃采集和关键链路日志。
2. 热更 DLL/AOT 清单校验。
3. UIConfig/配置/场景自动检查。
4. 断线重连和频道超时兜底。
5. 首测功能入口治理。

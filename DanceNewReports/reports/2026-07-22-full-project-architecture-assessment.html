Executive Summary

这是一个功能面已经铺开、底层技术路线明确，但验证成熟度仍落后于源码体量的研发后期项目。 可以进入 7/22 - 8/11 新周期，但必须将“继续开发”与“形成可复现首测候选包”分两条线管理。

  • 架构已经不是原型级。启动、热更、配置、网络、数据、UI、场景、玩法、SDK 与多类外圈系统均有成型实现；项目难点已经从“有没有功能”转为“跨系统是否稳定协同”。
  • 体验层的复杂度最高。UI 是非协议代码中最大模块，资源、预制体、角色外观、场景和渲染配置会共同决定首测的加载、内存、回归和性能压力。
  • 移动端性能尚不能判定为通过。项目具备三档 URP 质量配置和 SRP Batcher，但当前资料没有 Android 真机帧率、Draw Call、内存、温度或加载耗时报告；这不是“没有问题”，而是“尚未被证明”。
  • 当前最重要的不是继续扩大功能入口。应先解决干净候选基线、真实包热更、崩溃可观测、主链路弱网与性能基线，再决定哪些外圈功能进入首测。

关键解读：技术路线适合需要热更、内容频繁迭代的手游，但启动成功依赖 DLL/AOT、资源包、配置、程序集入口与平台 SDK 同时一致。编辑器运行顺利并不能替代 Android 真实包的启动和热更验证。

Technical choices

技术选型是合理的，但每项都带来了需要验收的代价

Unity 2022.3.57f1c1 + URP 14.0.11

目的：用通用渲染管线覆盖多质量档、移动端和角色/场景表现。
代价：质量档、Shader 变体、阴影、贴图和移动端 GPU 压力必须用真机数据校准。

HybridCLR 热更新 + 多程序集

目的：让业务 DLL 在资源更新后快速迭代。
代价：运行时装配点较多；AOT 元数据、DLL 清单、反射调用与包体配置要保持同一版本。

AssetBundle / StreamingAssets + 对象池

目的：将资源更新、首包和运行时加载从代码中拆开，UI 通过对象池复用。
代价：资源路径、Prefab、图集、配置和包清单的一致性变成首测 P0 级工作。

Luban 配置 + Protobuf 协议扩展

目的：配置驱动内容、按业务拆分服务端消息。
代价：生成代码体量很大，不能用总代码行数代表功能进度；导表、版本和服务端协议需要自动校验。

事件总线 + Singleton 数据管理器

目的：快速拆开 UI、数据、网络、场景间的直接调用。
代价：事件编号、监听生命周期、切场景和重连后的状态清理会成为隐性耦合点。

Cinemachine / Timeline / 第三方 SDK

目的：承载镜头演出、婚礼/拍照体验、聊天语音、渠道能力与数据采集。
代价：功能上线会同时依赖设备权限、平台回调、资源表现、多人状态与埋点口径。

UI 是最大的手写业务面

UI 共 68,267 行、387 个 C# 文件，占剔除协议生成代码后的约 43%。它直接接触入口、加载、数据、网络返回和资源，因此也是改动扩散与回归压力最大的区域。

协议文件很大，不等于功能就成熟

Proto 有 128,652 行，约占全部 C# 的 45%。它反映了服务端交互面较广，但首测是否可用仍取决于注册、网络状态、服务端环境、UI 回调和真实流程。

Event and state flow

事件系统让开发更快，也把“状态不同步”变成主要排查成本

静态扫描到 1000+ 个消息标记；数量说明覆盖面，不代表线上事件次数。

1. 服务端消息

NetworkManager 负责连接与序列化；18 个 ProtoExtensions 按登录、频道、房间、战斗、好友、活动等业务处理回包。

2. 业务状态

DataManager 及其 24 个文件承载玩家、房间、背包、好友、活动、邮件、任务等长期状态和缓存。

3. 事件广播

Core 和 SDK 分别封装事件门面；业务层以整型事件 ID 加泛型参数广播，支持 0 到 4 个参数的调用。

4. UI / 场景响应

UIManager、SceneManager、GamePlay 等订阅事件更新页面、加载场景、刷新角色或驱动战斗表现。

5. 玩家再次操作

UI 触发请求或事件，进入网络层并等待回包。断线、重复回包、切后台和切场景是此闭环最容易断开的地方。

架构判断：事件总线与 Singleton 对快速做功能很合适，且静态 AddListener / RemoveListener 数量接近（559 / 556）是一个正向信号；但它只是代码文本计数，不能证明没有运行时漏监听。首测应为启动、登录、频道、房间、战斗、返回、断线重连建立事件追踪与状态断言。

实现方式：UIManager 在运行时创建 UIRoot、EventSystem 与相机分层，按 UIConfig 控制底层、普通层、弹窗和顶层；加载使用 PoolManager.Spawn，关闭时根据配置隐藏或 DeSpawn。这套机制已具备通用 UI 框架的形态。

必须验证的地方：UIConfig、Prefab 路径、图集、动态入口和对象池不是独立问题。一个页面打不开可能来自配置、资源包、网络数据或事件顺序，所以需要以“入口到返回”的链路回归，而不是逐页点亮。

渲染技术与性能压力：工程准备存在，性能验收证据尚缺

项目设置证明“设计了三档”，但没有代替真实设备上的帧率、内存和发热结果。

补充：Android 默认质量索引为 Balanced。三档均启用 SRP Batcher，关闭 Dynamic Batching；质量资产还开启 Adaptive Performance 标记。

渲染与体验链路中已见的技术点

  • 角色外观：角色换装直接操作 SkinnedMeshRenderer；多人同场、换装和材质切换要看 CPU、GPU 与内存峰值。
  • 战斗表现：BattleRenderManager 按帧更新场上角色，并管理创建队列与销毁；多人战斗和婚礼演出是性能采样重点。
  • 镜头演出：Cinemachine 与 Timeline 用于广场、拍照和演出镜头，需验证切场景、回前台和镜头资源释放。
  • UI 特效：奖励界面等使用 ParticleSystem；UI 叠层、透明过绘、图集拆分和动画密度都需要在中低端机上实测。

性能风险判断：Android 默认 Balanced 已开启 HDR、2x MSAA、4 级阴影级联和 50 米阴影距离；项目启动目标帧率为 120，且所有质量档的 MipMap Streaming 均关闭。与此同时，源资源中有大量 2048 贴图、角色蒙皮、战斗角色与演出镜头。SRP Batcher 能减轻一部分 CPU 提交压力，但不能替代对 GPU、过绘、纹理内存、加载尖峰和温度的真机采样。当前资料未发现 Unity Profiler、Frame Debugger、Android GPU/内存或设备分档报告，因此性能状态应评为未验证。

Module maturity

功能与系统成熟度：源码覆盖不等于可以全量开放

“源码面”描述实现存在程度；“验证状态”描述本地资料中是否有足够交付证据。

启动、热更与资源更新源码面：高验证：缺关键证据

GameLaunch → AssetBundle → HybridCLR → GameMain 链路完整。首测前仍需 Android 包证明 DLL、AOT、配置与资源包在真实安装路径中一致。

配置与业务数据源码面：高验证：需生命周期回归

Luban 表、JSON、ConfigManager 与多类 DataManager 已形成数据中台。重点不在“有没有表”，而在切号、重登、回包重复、切场景后的缓存是否一致。

登录、频道、广场源码面：高验证：弱网未证

登录、创角、频道、广场路径均有 UI、网络和场景实现；频道 loading 的超时、失败返回与断线恢复仍应作为首测主链路。

房间、战斗与结算源码面：高验证：多人真机待补

房间创建、加入、准备、进入战斗、结算均见协议和业务实现。需要至少双人、多轮、断线/后台切换下的稳定回归，而不是单人编辑器路径。

UI、背包、设置、邮件、任务源码面：高首测：限范围

入口与页面覆盖较广，是首测体验可用性的组成部分；任务跳转、红点、活动入口要和首测白名单一致，避免带出未开放系统。

好友、聊天、家族、社交源码面：高家族：条件阻塞

好友和聊天具备实现基础；FamilyProtoExtensions 初始化在 NetworkManager 中被注释，而家族 UI 仍存在，因此家族应默认隐藏或完成专门联调后再开。

活动、商店、支付、经济源码面：高首测：严格控开

活动、商城、礼包、抽奖等代码和配置面广，依赖服务端、运营配置与 SDK。首次技术测试应关闭真实支付，入口只按验证目标逐项打开。

角色、换装、拍照、婚礼、心动房源码面：高表现与依赖重

这些系统叠加角色资源、镜头、多人同步、相册/权限或复杂演出，代码存在并不足以说明可面向普通测试用户开放，建议灰度或隐藏。

SDK、埋点、崩溃与运营观测源码面：部分到位验证：不足

埋点与渠道/IM SDK 已接入；Bugly 初始化在启动处被注释。没有崩溃回传、构建标识和主链路埋点，就无法快速解释首测失败来源。

自动化、构建与性能质量源码面：薄弱验证：缺失

工程包含 Unity Test Framework，但扫描未发现 NUnit、UnityTest、TestFixture 或 Assert 用例；也未发现新近真机构建、性能采样和自动构建验收资料。

对这组数字应怎样理解

不是“代码很差”的结论。项目有清晰的程序集、配置和协议拆分，事件监听的添加/移除文本计数也相近。

但它说明质量控制更多依赖人工。在热更、多资源、多入口、多人在线项目中，人工回归无法稳定覆盖每次资源/配置/程序集组合变化。

优先级：先给出包前静态检查、候选包启动日志、核心链路冒烟和崩溃/埋点校验，再考虑大规模单元测试补齐。

当前应被显式管理的风险，而不是“以后再看”的技术债

  • 01 首测要覆盖的 Android 最低机型、目标帧率和可接受的内存/加载/温度阈值是什么？
  • 02 哪个 revision 才是唯一候选包？冲突、资源版本、配置版本与服务端环境如何绑定？
  • 03 家族、拍照、婚礼、心动房、活动、商店和支付，哪些是首测必须开放，哪些必须隐藏？
  • 04 Bugly 或替代崩溃平台什么时候在真包验证？出现卡 loading 时用什么日志能区分资源、网络、服务端和 UI？
  • 05 三档画质是由设备自动选择、用户手动选择，还是统一默认？高画质是否会意外落到中低端 Android？
  • 06 服务端测试环境、域名、渠道 SDK、支付沙箱、权限与数据隔离是否已经具备可演练的配置？

证据限制：资源文件体量不是最终包体、下载量、显存或运行时内存；静态代码计数不是缺陷数量；未发现新增 Android 构建、Profiler 或真机测试资料，表示这些能力在当前资料中未被证明，而不是直接判定失败。
